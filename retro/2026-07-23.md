# 회고 — 2026-07-23

Claude Design v2 목업(사이드바 셸 + 카드 그리드) 기반 데스크톱 리디자인, 쓴템 탭 정리, 세션 보안 강화, 문서 동기화까지 마친 날의 회고.

## 나의 회고

**오늘 가장 힘들었던 부분은?**
디자인 의도를 말로 전달하는 게 생각보다 어려웠다. 목업 이미지와 실제 결과물 사이의 미묘한 차이(배경 대비, 여백, 버튼 배치)를 스크린샷에 손으로 표시해가며 여러 번 짚어줘야 겨우 맞아떨어졌고, 그 과정에서 시간도 토큰도 예상보다 훨씬 많이 들었다.

**에이전트를 쓰면서 고민되는 지점은?**
작업 순서를 어떻게 잡아야 할지 아직 감이 안 잡힌다. 디자인처럼 눈으로 확인해야 하는 작업을 큼직하게 한 번에 맡기는 게 나은지, 아니면 화면 하나하나 쪼개서 중간중간 확인시켜가며 진행하는 게 토큰·시간을 덜 쓰는 방법인지 판단이 안 선다.

## Claude의 회고

**오늘 가장 힘들었던 부분은?**
정적 목업(Claude Design 파일)을 실제 반응형 코드로 옮기는 과정에서, "색상 값이 같다"와 "레이아웃이 같아 보인다"는 완전히 다른 문제였다. 카테고리 칩 배경 누락, 폼 입력창이 카드와 같은 흰색이라 안 보이는 문제, 저장/삭제 버튼을 flex 비율로 나눴을 때 특정 창 폭에서만 필드가 줄바꿈되는 문제 — 세 가지 다 스크린샷만 봐서는 "대충 맞아 보이는데?" 싶었고, 실제로 `getBoundingClientRect()`로 폭을 재보고 나서야 정확한 원인이 잡혔다. 시각적 검증을 스크린샷 눈대중에만 의존하면 이런 종류의 버그를 계속 놓칠 것 같다.

**막혔을 때 나를 가장 도와준 규율은?**
"눈으로 보지 말고 재라." 사용자가 준 스크린샷만 보고 있었으면 "여백이 있으니 되어야 하는데?"만 반복했을 것이다. 실제 DOM에서 컴퓨티드 스타일과 좌표를 직접 찍어보고 나서야 어디서 얼마나 부족한지가 숫자로 나왔다.

**완성까지, 지금 가장 겁나는 부분은 무엇인가?**
사용자가 오늘 지적한 것처럼, 디자인 반영 작업은 "말로 설명 → 코드로 구현 → 스크린샷으로 확인 → 다시 설명"의 반복이 다른 작업보다 유독 길다. 이 루프를 줄이려면 처음부터 화면별로 잘게 쪼개서 매번 실제 브라우저로 확인하고 넘어가야 하는데, 그럼 지금보다 작업 단위를 훨씬 작게 나눠야 한다 — 그게 오히려 전체 시간을 늘리는 건 아닐지, 아직 균형을 못 잡았다.
