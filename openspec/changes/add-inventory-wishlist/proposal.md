## Why

'이미 있어' 앱은 아직 물품 관리 기능이 없다. 사용자가 이미 가지고 있는 물건(보유템)과 사고 싶은 물건(위시리스트)을 각각 관리하고, 위시리스트 항목을 구매한 뒤 보유템으로 자연스럽게 넘길 수 있어야 한다. 특히 위시리스트→보유템 전환 시 두 목록 중 하나에만 반영된 상태(중복 또는 유실)가 생기면 데이터 신뢰도가 깨지므로, 이 전환은 반드시 원자적으로 처리되어야 한다.

## What Changes

- 보유템(Owned Item) capability 신설: 이름·카테고리(자유 입력)·수량·구매일·사용기한(선택)·메모·상태(미개봉/사용중/다 씀) 필드를 가진 물품을 등록·조회·수정·삭제
- 위시리스트(Wishlist) capability 신설: 사려는 물품을 등록·조회·수정·삭제, '구매완료' 처리 액션 제공
- 위시리스트 '구매완료' 처리는 단일 트랜잭션으로 위시리스트 항목 삭제 + 보유템 신규 생성을 동시에 반영 (부분 실패 시 전체 롤백)
- 두 capability 모두 목록 조회 시 사용기한 임박순 정렬, 각 항목에 D-day 표시, 이름·카테고리 텍스트 검색 지원
- 두 capability 모두 로그인한 사용자만 접근 가능하며, 각 사용자는 자신이 등록한 데이터만 조회·수정·삭제 가능 (다른 사용자의 데이터에는 접근 불가)

### 리디자인 확장분 (Claude Design 목업 기반, 후속 라운드)
- 회원가입에 닉네임(필수, 2~20자), 약관동의 체크박스(필수) 추가, `user_metadata.nickname`으로 저장
- 비밀번호 찾기: `/forgot-password`에서 재설정 메일 요청 → `/reset-password`에서 새 비밀번호 설정까지 실동작하는 전체 플로우 신설
- 위시리스트 항목에 참고 링크(선택) 필드 추가, 위시리스트 항목 수정 화면(`/wishlist/[id]`) 신설 — 이전에는 등록·삭제만 가능했음
- 보유템/위시리스트 등록·수정 폼과 목록 필터의 카테고리 입력을 자유 텍스트에서 "기존 사용 카테고리 칩 선택 + 새 카테고리 추가" 방식으로 전환 (`list_owned_categories`/`list_wishlist_categories` RPC 신설). 저장되는 카테고리 값의 데이터 형식·검증 규칙은 변경 없음(여전히 자유 텍스트 문자열)
- 보유템 상태(미개봉/사용중/다 씀) 입력·표시를 드롭다운에서 3단계 세그먼트 스텝퍼 UI로 전환 (동작·값 자체는 변경 없음)
- 디자인 토큰(색상: 틸 그린 #00a090 액센트)과 로고 에셋을 Claude Design 목업 기준으로 교체, 사용기한 임박도 색상 구분(D-day 배지)은 기존 스펙대로 유지

### 쓴템 탭 + 세션 보안 확장분 (후속 라운드)
- 보유템에 `used_up_at`(다 씀으로 바뀐 날짜) 필드 추가. 상태가 "다 씀"으로 바뀌는 시점에만 오늘 날짜로 기록되고, 그 외 상태로 바뀌면(되돌리기 포함) 항상 null로 초기화된다
- "쓴템" 탭(`/used`) 신설: 다 씀 상태인 보유템만 모아 보여주고, 되돌리기 버튼으로 즉시 "사용중" 상태로 있템 목록에 복귀시킬 수 있다
- 있템 목록 정렬 규칙 정식화(`sortOwnedItemsForList`): 사용기한 임박순 → 사용기한 없는 항목은 등록 최근순 → 다 씀 항목은(토글로 노출했을 때) 정렬과 무관하게 항상 최하단, 자기들끼리는 상태 변경 최근순. 기본적으로 다 씀 항목은 있템 목록에서 제외되며, "다 쓴 것도 보기" 토글로만 노출된다
- 세션/계정 간 데이터 격리 강화: 로그아웃 시 클라이언트 상태·`localStorage`/`sessionStorage`(이메일 저장 값 제외) 전체 정리 후 전체 페이지 새로고침, 세션 만료 시 에러 대신 로그인 화면으로 리다이렉트, 브라우저 뒤로가기(bfcache) 복원 시 강제 재검증, 응답에 `Cache-Control: no-store` 강제 — 같은 브라우저에서 A 계정 로그아웃 후 B 계정으로 로그인해도 A의 데이터가 잔존하지 않도록 함

### 데스크톱 리디자인 v2 확장분 (Claude Design v2 목업 기반, 후속 라운드)
- 상단 고정 헤더+세로 리스트 구조를 좌측 고정 사이드바 셸(데스크톱)/상단 가로 바(모바일) + 카드 그리드 구조로 전면 전환
- 로그인/회원가입 화면을 좌측 틸 그라디언트 히어로 패널(캐치프레이즈·장점 3가지) + 우측 폼의 2단 카드로 재구성, 로그인↔회원가입 전환은 하단 텍스트 링크로 변경
- 위시 등록/수정 화면의 카테고리 칩이 위시 카테고리뿐 아니라 있템 카테고리도 함께 보여주도록 확장(합집합), 위시 "구매" 버튼을 "샀어요 · 있템으로"로 개칭하고 카드 하단 전체너비 버튼으로 이동
- 폼 카드 배경/입력창 대비 개선(`--input-bg` 토큰 신설), 저장+삭제 버튼을 같은 줄에 나란히 배치하는 공용 패턴(`secondaryAction` prop) 도입
- 회원가입 후 이메일 인증 대기 상태로 로그인 화면에 돌아올 때 발생하던 이중 리다이렉트(사이드바 오표시 버그) 수정, 로그인 화면에 가입 확인 안내 배너 + 이메일 자동 채움 추가

### 코드 리뷰 결함 수정 확장분 (후속 라운드)
- 보유템 수정 시 이미 "다 씀"인 항목을 다른 필드만 고쳐 재저장해도 `used_up_at`이 오늘 날짜로 덮어써지던 문제 수정 — 수정 경로 전용 `deriveUsedUpAtForUpdate`로 이전 상태를 함께 고려
- 보유템 수정 폼의 메모 입력이 리디자인 과정에서 단일 라인으로 퇴화해 줄바꿈이 유실되던 문제를 복원(사용기한과 나란히 배치되는 레이아웃은 유지)
- 저장 버튼과 삭제 버튼(`secondaryAction`)이 서로 다른 `<form>`임에도 저장 폼 안에 중첩되어 있던 유효하지 않은 HTML 구조를 형제 요소 + `form` 속성 기반 제출로 수정
- 비밀번호 재설정 리다이렉트가 요청의 `Host` 헤더를 검증 없이 신뢰하던 문제를 `NEXT_PUBLIC_SITE_URL` 기반으로 수정(프로덕션에서 미설정 시 fail-closed)
- 비밀번호 재설정 요청 실패 메시지가 필터링 없이 노출되던 문제를 로그인 화면과 공유하는 `lib/auth-errors.ts`로 수정
- 회원가입 직후 이메일이 URL 쿼리스트링에 노출되던 문제를 60초 만료 쿠키 기반 전달로 수정
- 사이드바와 세션 미들웨어가 "인증 예외 경로"를 서로 다른 규칙으로 판정하던 문제를 `lib/auth-routes.ts`의 단일 판정 함수로 통합

## Capabilities

### New Capabilities
- `inventory`: 로그인한 사용자가 자신의 보유템을 등록·조회·수정·삭제하고, 사용기한 임박순 정렬/D-day 표시/이름·카테고리 검색을 제공하는 capability
- `wishlist`: 로그인한 사용자가 자신의 위시리스트 항목을 등록·조회·삭제하고, '구매완료' 처리를 통해 보유템으로 원자적으로 전환하는 capability

### Modified Capabilities
(해당 없음 - 그린필드 기능이며 기존 spec이 없음)

## Impact

- 신규 백엔드 API 및 DB 스키마(보유템 테이블/컬렉션, 위시리스트 테이블/컬렉션) 필요
- 인증된 사용자 컨텍스트(user id)를 전제로 한 인가 로직(리소스 소유자 검증) 필요 — 회원가입/로그인 화면 자체의 설계는 이 변경 범위 밖이며, 로그인 세션에서 사용자 식별자를 얻을 수 있다고 가정한다
- 위시리스트→보유템 전환을 위한 트랜잭션 처리 계층 필요 (백엔드 스택별 구현 방식은 design.md에서 비교)
- 목록 API에 정렬·검색 파라미터 추가, 프론트엔드 목록 UI에 D-day 표시 및 검색 입력 추가
