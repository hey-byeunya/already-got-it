## Why

'이미 있어' 앱은 아직 물품 관리 기능이 없다. 사용자가 이미 가지고 있는 물건(보유템)과 사고 싶은 물건(위시리스트)을 각각 관리하고, 위시리스트 항목을 구매한 뒤 보유템으로 자연스럽게 넘길 수 있어야 한다. 특히 위시리스트→보유템 전환 시 두 목록 중 하나에만 반영된 상태(중복 또는 유실)가 생기면 데이터 신뢰도가 깨지므로, 이 전환은 반드시 원자적으로 처리되어야 한다.

## What Changes

- 보유템(Owned Item) capability 신설: 이름·카테고리(자유 입력)·수량·구매일·사용기한(선택)·메모·상태(미개봉/사용중/다 씀) 필드를 가진 물품을 등록·조회·수정·삭제
- 위시리스트(Wishlist) capability 신설: 사려는 물품을 등록·조회·삭제, '구매완료' 처리 액션 제공
- 위시리스트 '구매완료' 처리는 단일 트랜잭션으로 위시리스트 항목 삭제 + 보유템 신규 생성을 동시에 반영 (부분 실패 시 전체 롤백)
- 두 capability 모두 목록 조회 시 사용기한 임박순 정렬, 각 항목에 D-day 표시, 이름·카테고리 텍스트 검색 지원
- 두 capability 모두 로그인한 사용자만 접근 가능하며, 각 사용자는 자신이 등록한 데이터만 조회·수정·삭제 가능 (다른 사용자의 데이터에는 접근 불가)

## Capabilities

### New Capabilities
- `inventory`: 로그인한 사용자가 자신의 보유템을 등록·조회·수정·삭제하고, 사용기한 임박순 정렬/D-day 표시/이름·카테고리 검색을 제공하는 capability
- `wishlist`: 로그인한 사용자가 자신의 위시리스트 항목을 등록·조회·삭제하고, '구매완료' 처리를 통해 보유템으로 원자적으로 전환하는 capability

### Modified Capabilities
(해당 없음 - 그린필드 기능이며 기존 spec이 없음)

## Impact

- 신규 백엔드 API 및 DB 스키마(보유템 테이블/컬렉션, 위시리스트 테이블/컬렉션) 필요
- 인증된 사용자 컨텍스트(user id)를 전제로 한 인가 로직(리소스 소유자 검증) 필요 — 회원가입/로그인 화면 자체의 설계는 이 변경 범위 밖이며, 로그인 세션에서 사용자 식별자를 얻을 수 있다고 가정한다
- 위시리스트→보유템 전환을 위한 트랜잭션 처리 계층 필요 (백엔드 스택별 구현 방식은 design.md에서 비교)
- 목록 API에 정렬·검색 파라미터 추가, 프론트엔드 목록 UI에 D-day 표시 및 검색 입력 추가
