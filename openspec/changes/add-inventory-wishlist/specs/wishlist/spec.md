## ADDED Requirements

### Requirement: 인증된 사용자만 접근
시스템은 위시리스트 관련 모든 API(조회/등록/삭제/구매완료 처리)를 로그인한 사용자에게만 허용해야 한다(SHALL). 로그인하지 않은 요청은 반드시(MUST) 거부되어야 한다.

#### Scenario: 비로그인 사용자의 접근 차단
- **WHEN** 로그인하지 않은 클라이언트가 위시리스트 목록 조회, 등록, 삭제 또는 구매완료 처리 API를 호출한다
- **THEN** 시스템은 요청을 거부하고 인증 필요 오류를 반환한다

### Requirement: 사용자별 데이터 격리
시스템은 각 사용자가 자신이 등록한 위시리스트 항목만 조회·삭제·구매완료 처리할 수 있도록 SHALL 보장해야 한다. 다른 사용자의 위시리스트 항목에는 어떤 방식으로도 접근할 수 없어야 한다(MUST NOT).

#### Scenario: 타인의 위시리스트 항목 접근 시도 차단
- **WHEN** 사용자 A가 사용자 B 소유의 위시리스트 항목 ID로 조회, 삭제 또는 구매완료 처리를 요청한다
- **THEN** 시스템은 해당 요청을 거부하고 데이터를 노출하지 않는다

#### Scenario: 자신의 위시리스트 목록만 조회됨
- **WHEN** 사용자가 위시리스트 목록을 조회한다
- **THEN** 시스템은 해당 사용자가 등록한 위시리스트 항목만 반환하고 다른 사용자의 항목은 포함하지 않는다

### Requirement: 위시리스트 항목 등록 및 삭제
시스템은 이름(필수), 카테고리(자유 입력, 선택), 메모(선택)를 입력받아 위시리스트 항목을 등록하고, 사용자가 자신의 위시리스트 항목을 삭제할 수 있도록 SHALL 지원해야 한다.

#### Scenario: 위시리스트 항목 등록
- **WHEN** 사용자가 이름을 입력해 위시리스트 등록을 요청한다
- **THEN** 시스템은 카테고리와 메모가 없어도 새 위시리스트 항목을 생성한다

#### Scenario: 위시리스트 항목 삭제
- **WHEN** 사용자가 자신의 위시리스트 항목 삭제를 요청한다
- **THEN** 시스템은 해당 항목을 목록에서 제거하고 이후 조회에서 반환하지 않는다

### Requirement: 이름 및 카테고리 검색
시스템은 사용자가 이름 또는 카테고리에 포함된 텍스트로 자신의 위시리스트 항목을 검색할 수 있도록 SHALL 지원해야 한다.

#### Scenario: 이름으로 검색
- **WHEN** 사용자가 검색어를 입력해 위시리스트 목록을 조회한다
- **THEN** 시스템은 이름에 검색어가 포함된 위시리스트 항목만 반환한다

#### Scenario: 카테고리로 검색
- **WHEN** 사용자가 카테고리 값과 일치하는 검색어를 입력한다
- **THEN** 시스템은 카테고리에 검색어가 포함된 위시리스트 항목만 반환한다

### Requirement: 구매완료 처리 시 보유템으로의 원자적 전환
사용자가 위시리스트 항목을 '구매완료' 처리하면, 시스템은 단일 트랜잭션 내에서 (1) 해당 위시리스트 항목을 삭제하고 (2) 그 정보를 이어받은 새 보유템을 생성해야 한다(SHALL). 두 동작 중 하나만 반영된 중간 상태가 존재해서는 안 된다(MUST NOT). 생성되는 보유템의 이름·카테고리는 위시리스트 항목에서 이어받고, 구매일은 구매완료 처리 시각의 날짜로 자동 설정되며, 상태는 기본값인 "미개봉"으로 설정되어야 한다(MUST). 위시리스트에는 수량 개념이 없으므로, 생성되는 보유템의 수량은 기본값 1로 설정되어야 한다(MUST). 메모가 있었다면 그대로 이어받는다.

#### Scenario: 구매완료 처리 성공
- **WHEN** 사용자가 자신의 위시리스트 항목에 대해 구매완료 처리를 요청한다
- **THEN** 시스템은 해당 위시리스트 항목을 삭제하고, 동일한 트랜잭션 내에서 이름·카테고리·메모를 이어받고 구매일은 오늘 날짜, 상태는 "미개봉", 수량은 1인 보유템을 새로 생성하며, 두 결과가 함께 커밋된다

#### Scenario: 구매완료 처리 중 실패 시 전체 롤백
- **WHEN** 구매완료 처리 과정에서 보유템 생성 또는 위시리스트 항목 삭제 중 하나라도 실패한다
- **THEN** 시스템은 트랜잭션 전체를 롤백하여 위시리스트 항목을 삭제 이전 상태로 유지하고 보유템도 생성되지 않은 상태로 유지한다

#### Scenario: 구매완료 처리 후 응답
- **WHEN** 구매완료 처리가 성공적으로 커밋된다
- **THEN** 시스템은 응답으로 새로 생성된 보유템의 정보(ID 포함)를 반환한다

#### Scenario: 이미 처리되었거나 존재하지 않는 항목에 대한 재요청
- **WHEN** 사용자가 이미 구매완료 처리되어 삭제된(또는 존재하지 않는) 위시리스트 항목 ID로 구매완료 처리를 다시 요청한다
- **THEN** 시스템은 요청을 거부하고 항목을 찾을 수 없다는 오류를 반환하며, 보유템을 중복 생성하지 않는다
