## ADDED Requirements

### Requirement: 인증된 사용자만 접근
시스템은 보유템 관련 모든 API(조회/등록/수정/삭제)를 로그인한 사용자에게만 허용해야 한다(SHALL). 로그인하지 않은 요청은 반드시(MUST) 거부되어야 한다.

#### Scenario: 비로그인 사용자의 접근 차단
- **WHEN** 로그인하지 않은 클라이언트가 보유템 목록 조회 또는 등록/수정/삭제 API를 호출한다
- **THEN** 시스템은 요청을 거부하고 인증 필요 오류를 반환한다

### Requirement: 사용자별 데이터 격리
시스템은 각 사용자가 자신이 등록한 보유템만 조회·수정·삭제할 수 있도록 SHALL 보장해야 한다. 다른 사용자의 보유템에는 어떤 방식으로도 접근할 수 없어야 한다(MUST NOT).

#### Scenario: 타인의 보유템 접근 시도 차단
- **WHEN** 사용자 A가 사용자 B 소유의 보유템 ID로 조회, 수정 또는 삭제를 요청한다
- **THEN** 시스템은 해당 요청을 거부하고 데이터를 노출하지 않는다

#### Scenario: 자신의 보유템 목록만 조회됨
- **WHEN** 사용자가 보유템 목록을 조회한다
- **THEN** 시스템은 해당 사용자가 등록한 보유템만 반환하고 다른 사용자의 보유템은 포함하지 않는다

### Requirement: 보유템 등록
시스템은 이름, 카테고리(자유 입력), 수량, 구매일, 사용기한(선택), 메모, 상태를 입력받아 보유템을 등록할 수 있어야 한다(SHALL). 이름·카테고리·수량·구매일·상태는 필수이며, 사용기한과 메모는 선택이다. 수량을 입력하지 않으면 기본값 1로 등록되어야 한다(MUST). 상태 값은 미개봉/사용중/다 씀 중 하나여야 하며(MUST), 그 외 값은 거부되어야 한다.

#### Scenario: 필수 항목만으로 등록
- **WHEN** 사용자가 이름, 카테고리, 구매일, 상태를 입력해 보유템 등록을 요청한다
- **THEN** 시스템은 사용기한과 메모 없이, 수량은 기본값 1로 새 보유템을 생성한다

#### Scenario: 수량 미입력 시 기본값 적용
- **WHEN** 사용자가 수량을 입력하지 않고 보유템 등록을 요청한다
- **THEN** 시스템은 수량을 1로 설정해 보유템을 생성한다

#### Scenario: 사용기한 포함 등록
- **WHEN** 사용자가 사용기한을 포함해 보유템 등록을 요청한다
- **THEN** 시스템은 입력된 사용기한을 저장하고, 이후 목록 조회 시 정렬 및 D-day 계산에 사용한다

#### Scenario: 허용되지 않은 상태 값 거부
- **WHEN** 사용자가 상태 값으로 미개봉/사용중/다 씀 이외의 값을 입력해 등록 또는 수정을 요청한다
- **THEN** 시스템은 요청을 거부하고 유효성 오류를 반환한다

#### Scenario: 허용되지 않은 수량 값 거부
- **WHEN** 사용자가 수량으로 0 이하이거나 정수가 아닌 값을 입력해 등록 또는 수정을 요청한다
- **THEN** 시스템은 요청을 거부하고 유효성 오류를 반환한다

### Requirement: 보유템 수정 및 삭제
시스템은 사용자가 자신의 보유템의 이름·카테고리·수량·구매일·사용기한·메모·상태를 수정하거나 삭제할 수 있도록 SHALL 지원해야 한다.

#### Scenario: 상태 변경
- **WHEN** 사용자가 자신의 보유템 상태를 미개봉에서 사용중으로 변경 요청한다
- **THEN** 시스템은 해당 보유템의 상태를 사용중으로 갱신한다

#### Scenario: 보유템 삭제
- **WHEN** 사용자가 자신의 보유템 삭제를 요청한다
- **THEN** 시스템은 해당 보유템을 목록에서 제거하고 이후 조회에서 반환하지 않는다

### Requirement: 사용기한 임박순 정렬과 D-day 표시
시스템은 보유템 목록을 사용기한이 임박한 순서로 정렬해야 하며(SHALL), 각 항목에는 오늘 기준 D-day를 함께 표시해야 한다(MUST). 사용기한이 없는 항목은 사용기한이 있는 항목들보다 뒤에 표시되어야 한다.

#### Scenario: 사용기한 임박순 정렬
- **WHEN** 사용자가 사용기한이 서로 다른 여러 보유템을 목록 조회한다
- **THEN** 시스템은 사용기한이 가장 임박한(가까운) 항목부터 순서대로 반환한다

#### Scenario: 사용기한 없는 항목은 목록 뒤로 정렬
- **WHEN** 사용기한이 설정된 항목과 설정되지 않은 항목이 함께 조회된다
- **THEN** 시스템은 사용기한이 없는 항목을 사용기한이 있는 모든 항목 뒤에 배치한다

#### Scenario: D-day 표시
- **WHEN** 사용기한이 오늘로부터 N일 남은 보유템을 조회한다
- **THEN** 시스템은 해당 항목에 D-N(N일 남음) 형식의 D-day 값을 함께 반환하며, 사용기한이 지난 경우 경과일 수를 표시한다

### Requirement: 이름 및 카테고리 검색
시스템은 사용자가 이름 또는 카테고리에 포함된 텍스트로 자신의 보유템을 검색할 수 있도록 SHALL 지원해야 한다.

#### Scenario: 이름으로 검색
- **WHEN** 사용자가 검색어를 입력해 보유템 목록을 조회한다
- **THEN** 시스템은 이름에 검색어가 포함된 보유템만 반환한다

#### Scenario: 카테고리로 검색
- **WHEN** 사용자가 카테고리 값과 일치하는 검색어를 입력한다
- **THEN** 시스템은 카테고리에 검색어가 포함된 보유템만 반환한다

#### Scenario: 검색 결과 없음
- **WHEN** 사용자의 검색어가 어떤 보유템의 이름·카테고리에도 포함되지 않는다
- **THEN** 시스템은 빈 목록을 반환한다
