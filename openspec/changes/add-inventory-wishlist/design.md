## Context

'이미 있어'는 개인용 물품 관리 앱으로, 이번 변경으로 보유템(inventory)과 위시리스트(wishlist) 두 capability를 처음 도입한다. 아직 백엔드가 존재하지 않는 그린필드 상태이므로 스택 자체를 선택해야 한다.

핵심 제약은 세 가지다.
- 모든 데이터는 로그인한 사용자 단위로 격리되어야 한다(다른 사용자의 데이터에 절대 접근 불가).
- 위시리스트 '구매완료' 처리는 위시리스트 삭제 + 보유템 생성을 하나의 원자적 단위로 처리해야 하며, 부분 반영(하나만 성공)이 생기면 안 된다.
- 목록은 사용기한 임박순 정렬(+ D-day), 이름/카테고리 검색을 지원해야 한다 — 관계형 정렬·필터 질의에 가까운 요구다.

## Goals / Non-Goals

**Goals:**
- 위 세 제약을 안전하고 검증 가능한 방식으로 만족하는 백엔드 스택을 선택하고 근거를 남긴다
- 사용자별 데이터 격리를 애플리케이션 로직뿐 아니라 가능하면 데이터 계층에서도 보장해, "코드 실수로 인한 데이터 유출" 위험을 줄인다
- 위시리스트→보유템 전환을 DB 트랜잭션 수준에서 원자적으로 구현할 방법을 확정한다

**Non-Goals:**
- 회원가입/로그인 UI 및 전체 인증 플로우(비밀번호 재설정, 소셜 로그인 연동 세부 등) 설계 — 인증 제공자가 로그인된 사용자 식별자(user id)를 제공한다는 전제만 사용한다
- 다중 리전/대규모 트래픽 대비 스케일링 설계 — 개인용 앱 규모를 전제로 한다
- 실시간 동기화(다른 기기 간 실시간 반영) — 요구사항에 없으므로 다루지 않는다

## Decisions

### 결정 1: 백엔드 스택 후보 비교

| 후보 | 인증 | 사용자별 격리 방식 | 원자적 트랜잭션 | 정렬/검색 적합성 | 운영 부담 |
|---|---|---|---|---|---|
| **Supabase (Postgres + Auth + RLS)** | 내장 Auth(이메일/OAuth), JWT 발급 | Row Level Security로 `auth.uid() = user_id`를 DB 레벨에서 강제 — 앱 코드 버그와 무관하게 격리 보장 | Postgres 트랜잭션/PL-pgSQL 함수(RPC)로 삭제+생성을 한 트랜잭션에 담아 커밋/롤백 | 관계형 쿼리 — `ORDER BY expiry_date NULLS LAST`, `ILIKE`/`pg_trgm` 검색이 표준 SQL로 자연스럽게 해결됨 | 매니지드형이라 인프라 운영 최소, 무료 티어로 개인 앱에 충분 |
| **Firebase (Firestore + Auth)** | 내장 Auth | Security Rules로 `request.auth.uid == resource.data.userId` 지정 가능 | Firestore 트랜잭션으로 다중 문서 원자적 갱신 가능하나 컬렉션 간 트랜잭션 코드를 직접 관리해야 함 | 문서형 DB라 범위 정렬(사용기한 임박순)과 다중 필드 OR 텍스트 검색에 약함 — 복합 인덱스 설계나 외부 검색 서비스(Algolia 등) 추가 필요 | 매니지드형, 운영 부담 적음 |
| **자체 구축 (NestJS + PostgreSQL + Prisma, 인증 직접 구현)** | JWT/세션을 직접 구현(Passport 등) | 모든 쿼리에 `WHERE user_id = :currentUser`를 앱 코드에서 매번 강제 — 실수로 누락 시 데이터 유출 위험이 DB가 아닌 코드 리뷰에 전적으로 의존 | Prisma `$transaction`으로 표준 SQL 트랜잭션 처리 가능 | Postgres 기반이라 정렬/검색은 Supabase와 동등 | 인증 시스템·서버 호스팅·배포·백업을 직접 운영해야 해 초기 구현·운영 비용이 가장 큼 |

**선택: Supabase (Postgres + Auth + Row Level Security)**

근거:
1. **격리 요구사항이 DB 레벨에서 강제된다.** "각자 자기 데이터만" 요구는 이 기능의 핵심 안전 요구사항인데, RLS를 쓰면 사용자별 격리가 애플리케이션 코드의 `WHERE` 절 실수에 의존하지 않고 정책으로 고정된다. 자체 구축안은 매 쿼리마다 사람이 조건을 빠뜨리지 않아야 하므로 회귀 위험이 더 크다.
2. **원자적 전환을 표준 SQL 트랜잭션/함수로 자연스럽게 구현 가능.** Postgres RPC 함수(`mark_wishlist_purchased(item_id)`) 안에서 `DELETE ... RETURNING` 후 `INSERT`를 한 트랜잭션으로 묶으면 부분 실패 시 자동 롤백된다. Firestore도 트랜잭션을 지원하지만 컬렉션 간 트랜잭션 로직을 애플리케이션에서 직접 조합해야 해 상대적으로 구현·검증 부담이 크다.
3. **정렬/검색 요구가 관계형 질의에 잘 맞는다.** 사용기한 임박순 정렬(NULL 처리 포함)과 이름/카테고리 텍스트 검색은 SQL의 `ORDER BY`, `ILIKE`/`pg_trgm` 인덱스로 바로 해결되며, Firestore처럼 별도 검색 서비스를 얹을 필요가 없다.
4. **인증을 별도로 구현할 필요가 없다.** Supabase Auth가 로그인/세션/JWT 발급을 제공하므로, 이 변경의 Non-Goal인 "인증 플로우 자체 설계"를 실제로 범위 밖에 둘 수 있다. 자체 구축안은 이 부분을 새로 만들어야 해 Non-Goal 전제가 깨진다.
5. **개인용 앱 규모에 맞는 운영 비용.** 매니지드 서비스라 서버·백업 운영 부담이 없고, 무료 티어로 시작 가능해 초기 구현 속도가 가장 빠르다.

### 결정 2: 위시리스트→보유템 전환은 단일 DB 트랜잭션(RPC 함수)으로 구현

두 개의 별도 API 호출(위시리스트 삭제 API + 보유템 생성 API)로 나누지 않고, 하나의 Postgres 함수(RPC) 안에서 `DELETE`와 `INSERT`를 한 트랜잭션으로 실행한다. 애플리케이션 서버(또는 클라이언트)가 두 단계를 오케스트레이션하면 중간에 프로세스가 죽거나 네트워크가 끊겼을 때 부분 반영 상태가 남을 수 있기 때문에, 원자성을 DB 트랜잭션 경계 안으로 가져온다.

### 결정 3: 사용기한 정렬은 `NULLS LAST` + 인덱스로 처리

`expiry_date`에 인덱스를 두고 `ORDER BY expiry_date NULLS LAST`로 조회한다. D-day는 저장하지 않고 조회 시점에 `expiry_date - CURRENT_DATE`로 계산해 항상 최신 값을 반환한다(별도 배치/캐시 갱신이 필요 없음).

### 결정 4: 이름/카테고리 검색은 대소문자 무시 부분일치로 처리

`name ILIKE '%검색어%' OR category ILIKE '%검색어%'`를 기본으로 하고, 데이터량이 늘어날 경우 `pg_trgm` GIN 인덱스로 성능을 보강한다.

## Risks / Trade-offs

- **[위험] RLS 정책 설정 실수로 격리가 깨질 수 있음** → 마이그레이션에 RLS 정책을 함께 버전 관리하고, 두 capability 모두 "타인의 데이터 접근 차단" 시나리오를 통합 테스트로 고정해 회귀를 방지한다.
- **[위험] Supabase 벤더 종속** → RPC 함수와 RLS 정책은 표준 SQL/PLpgSQL로 작성해, 필요 시 다른 Postgres 호스팅으로 이전 가능하도록 유지한다(Supabase 전용 확장 기능은 최소로 사용).
- **[트레이드오프] RPC 함수 방식은 일반 REST/ORM 흐름보다 로직이 DB 쪽에 위치해 애플리케이션 코드만 보는 개발자에게는 가시성이 떨어질 수 있음** → 함수 정의를 저장소 내 마이그레이션 파일로 관리하고 tasks.md에서 별도 리뷰 대상으로 명시한다.
- **[위험] 구매완료 처리 중 재시도로 인한 중복 생성** → RPC 함수 내에서 대상 위시리스트 항목 존재 여부를 확인 후 처리하고, 존재하지 않으면 오류를 반환해 중복 보유템 생성을 방지한다(specs/wishlist/spec.md의 "이미 처리되었거나 존재하지 않는 항목" 시나리오 참고).

## Migration Plan

1. Supabase 프로젝트에 `owned_items`, `wishlist_items` 테이블과 `user_id` 컬럼(FK to auth.users) 생성
2. 두 테이블에 RLS 활성화 + `user_id = auth.uid()` 정책(조회/등록/수정/삭제) 적용
3. `expiry_date`, `(name, category)` 검색용 인덱스 생성
4. `mark_wishlist_purchased(wishlist_item_id)` PL-pgSQL 함수 작성(단일 트랜잭션으로 DELETE + INSERT, 존재하지 않으면 예외 발생)
5. 신규 기능이므로 기존 데이터 마이그레이션은 없음 — 새 스키마 배포 후 바로 사용 가능
6. 롤백 전략: 신규 테이블/함수이므로 문제가 생기면 해당 마이그레이션만 되돌리면 되고 기존 기능에 영향 없음

## Open Questions

- 카테고리를 자유 입력으로 유지하되, 추후 자동완성을 위한 카테고리 목록 캐싱이 필요할지는 사용 데이터가 쌓인 뒤 결정한다
- 사용기한이 지난 항목을 목록에서 별도로 강조 표시할지(예: 색상 경고) 여부는 이번 변경 범위에서는 D-day 값 표시로 충분하다고 보고, UI 세부 사항은 구현 단계에서 결정한다
